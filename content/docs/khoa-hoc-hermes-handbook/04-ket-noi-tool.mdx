## Vấn đề kinh doanh

Nếu Hermes chỉ “trả lời hay”, CEO vẫn phải tự làm phần còn lại.

Tự mở file, tự copy sang email, tự cập nhật sheet, tự kiểm tra web, tự gửi tài liệu, tự nhớ lịch chạy báo cáo. Đó là khác biệt giữa **AI chat** và **AI agent**. Bấm để so sánh:

<ChatAgentToggle04 />

<Callout>
  <p>Chat AI tạo nội dung. Agent có tool có thể làm việc trong hệ thống thật. Có rule + approval thì agent vận hành an toàn.</p>
</Callout>

## Tool là gì trong Hermes?

Model là bộ não. Hermes là hệ điều phối. Tool là *tay chân* — khả năng hành động.

Bấm từng tool quanh Hermes để xem nó làm được gì và dùng cho việc nào:

<ToolHub04 />

Câu nhớ nhanh:

```text
Không có tool: Hermes tư vấn.
Có tool: Hermes vận hành.
Có rule + approval: Hermes vận hành an toàn.
```

## Nguyên tắc kết nối: ít nhưng đúng

Đừng bật hết. Bắt đầu bằng 1–2 tool tạo ROI rõ nhất, theo từng giai đoạn.

Bấm từng chặng trên lộ trình để xem nên nối tool gì và vì sao:

## Telegram — giao diện CEO nên dùng đầu tiên

Ra lệnh bằng điện thoại, nhận kết quả ngay trong chat, duyệt nhanh.

Với Telegram, mỗi loại việc nên có một quyền hạn rõ. Bấm từng loại việc:

## File & folder — nơi agent làm việc an toàn nhất

Trước khi nối email hay sheet, cho Hermes một “không gian làm việc”.

Bấm từng folder để xem dùng vào việc gì:

<FolderTree04 />

<CopySnippets snippets={[
  { title: 'PROMPT LÀM VIỆC CÓ CẤU TRÚC', code: 'Đọc file `/inputs/demo-abc.md`, tạo email follow-up theo template trong `/templates/follow-up.md`, lưu kết quả vào `/outputs/follow-up-abc.md`. Chưa gửi email.' }
]} />

Bắt đầu bằng file/folder vì: dễ kiểm tra, ít rủi ro hơn email/database, tạo quy trình rõ, và dễ biến thành skill sau này.

## Gmail — bắt đầu bằng draft, chưa gửi

Email giá trị cao nhưng rủi ro cao. Sai một email có thể mất cơ hội bán hàng.

Kéo mức quyền email từ an toàn → rủi ro để thấy khuyến nghị:

<GmailLevels04 />

<CopySnippets snippets={[
  { title: 'PROMPT CÓ APPROVAL', code: 'Hãy chuẩn bị email gửi khách ABC. Sau khi draft xong, hỏi anh xác nhận bằng câu: “Anh có muốn em gửi email này không?” Chỉ gửi nếu anh trả lời rõ “gửi”.' }
]} />

Luôn yêu cầu Hermes tách rõ **fact** và **giả định**, ghi nguồn. Tự chấm 5 mục checklist research:

## Ma trận quyền hạn theo tool

Gom lại tất cả: tool nào được làm gì, ô nào cần CEO duyệt.

Mỗi ô là một mức quyền: **Cho phép**, **Cần duyệt**, **Không tự làm**. Bấm ô để xem giải thích.

<PermissionMatrix04 />

## Thiết kế approval flow

Quy trình để Hermes biết: lúc nào tự làm, lúc nào phải hỏi.

Không có approval flow → agent hỏi quá nhiều, hoặc tự làm quá nhiều. Bấm từng mức rủi ro:

<RiskFlow04 />

<CopySnippets snippets={[
  { title: 'CÂU RULE NÊN DÙNG TRONG MỌI AGENT', code: 'Em được tự xử lý việc rủi ro thấp. Với việc gửi ra ngoài, sửa dữ liệu quan trọng, hoặc phát sinh chi phí, em phải hỏi anh trước. Nếu không chắc mức rủi ro, hãy dừng và hỏi.' }
]} />

## Hermes Operator Challenge

**Mục tiêu:** thiết kế tool stack tối giản cho một use case, không bật tool theo cảm hứng.

### Output cần nộp

Tạo file:

```text
tool-approval-map.md
```

Trong đó chọn 1 use case và mapping tool/quyền hạn rõ ràng.

| Use case | Tool cần dùng | Hermes tự làm | Hermes chỉ draft | CEO phải duyệt | Rủi ro |
|---|---|---|---|---|---|
|  |  |  |  |  |  |

### Prompt copy dùng ngay

```text
Anh muốn thiết kế tool stack an toàn cho use case sau:
[use case]

Các tool/kênh anh có hoặc muốn dùng:
[Telegram, file, Gmail, Sheet, web/browser, CRM...]

Hãy đề xuất bảng:
- Tool nào thật sự cần
- Tool nào chưa cần bật
- Hermes được tự làm gì
- Hermes chỉ được draft gì
- Việc nào CEO bắt buộc duyệt
- Rủi ro nếu cấu hình sai
- Bài test đầu tiên để kiểm tra tool hoạt động
```

### Tiêu chuẩn hoàn thành

- [ ] Chọn được 1 use case cần tool thật.
- [ ] Biết tool nào cần bật trước, tool nào để sau.
- [ ] Có approval rule trước khi gửi email/tin nhắn/dữ liệu ra ngoài.
- [ ] Có workspace file/folder hoặc kênh Telegram rõ.
- [ ] Có bài test an toàn trước khi chạy thật.
